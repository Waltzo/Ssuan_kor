# 작혼 리치 마작 오버레이 어시스턴트 — 개발 계획

## 1. 개요

Windows 환경에서 **MahjongSoul(작혼)** 클라이언트 위에 투명 오버레이로 떠서, 화면을 캡처·인식하여 실시간으로 리치 마작 조언을 제공하는 프로그램.

### 설계 철학
> **"보여주는 툴"이 아니라 "결정해주는 툴"**

샹텐·우케이레 같은 정보는 플레이어가 이미 안다. 실전에서 쓰이려면 **결정 보조**가 핵심:
타패 추천 / 후리텐 경고 / 오시히키(押し引き) 판단이 빠지면 실전에서 안 쓴다.

### 제공 기능
- **타패 추천**: 어느 패를 버릴지 — 우케이레 매수 + 대기 형태의 질 기반
- **후리텐 경고**: 텐파이인데 후리텐이면 경고 (론 불가)
- **役なし 경고**: 텐파이인데 역이 없으면 경고 (리치/쯔모만 가능)
- **위험패 분석**: 현물·스지(중/편/우라/걸침)·노찬스·字牌 안전도
- **오시히키 판단**: 내 패 가치 vs 방총 리스크 → 밀기/접기 결론
- **베타오리 시퀀싱**: 접기로 결정 시 매 순목 가장 안전한 패 제시
- **리치 판단**: 리치 / 다마텐 / 안 침
- **부로 판단**: 폰·치를 할지 (속도 vs 멘젠 타점)
- **상대 역 예측**: 버림패 패턴으로 역 경향 추정 (보조지표)
- **점수상황 전략**: 올라스 점수차 기반 톱 목표 / 라스 회피 모드

### 결정된 사항
| 항목 | 선택 |
|------|------|
| 룰 | 리치 마작 (일본식) |
| 대상 클라이언트 | 작혼 (MahjongSoul) |
| 입력 방식 | 화면 캡처 + 패 인식 |
| 기술 스택 | Python |

## 4. 모듈별 상세

### 4.1 Capture Layer (`src/capture/`)
- `win32gui.FindWindow` / `EnumWindows`로 작혼 창 핸들·클라이언트 영역 좌표 획득
- `dxcam`으로 해당 영역만 캡처 (전체 화면 캡처 후 크롭보다 효율적)
- 캡처 루프는 별도 스레드, 2~5 fps. 프레임 해시 비교로 정지 화면이면 스킵
- **요구사항**: 작혼을 **창 모드**로 실행 권장 (전체화면 독점 모드는 오버레이가 가려질 수 있음). 창 모드 자동 감지·경고

### 4.2 Recognition Layer (`src/recognition/`)
패 인식이 이 프로젝트의 가장 어려운 부분. 단계적 접근.

**좌표 기반 슬롯 분할**
- 작혼 UI는 레이아웃이 일정 → 클라이언트 해상도별 프로파일로 각 요소 영역을 상대좌표(%)로 정의
  - 내 손패 (하단 13~14장) — **가장 쉬움, 평면·정면**
  - 4인 버림패(河) — 작고 일부 회전(리치 선언패·울린 패), 원근 있음
  - 부로(鳴き) 멘츠 — 각 플레이어 우측 (누가 누구에게서 울었는지 방향 포함)
  - 도라 표시패 — 패산 위
  - 게임 상태 텍스트 — 장풍/자풍, 점수, 본장·리치봉, 남은 패 수, 親/子
- 캘리브레이션 UI로 사용자가 처음 1회 영역 보정

**패 분류 (1차: 템플릿 매칭)**
- 작혼은 패 그래픽이 고정 → 34종 패 + 아카도라(빨강 5p/5s/5m) 템플릿 이미지를 `assets/templates/`에 저장
- 각 슬롯 크롭 → `cv2.matchTemplate`로 최고 점수 패 판정
- 회전된 패는 회전 보정 후 매칭

**패 분류 (선택적 2차: ML 모델)**
- 작혼 패 그래픽은 매우 정형적 → **템플릿 매칭으로 충분할 가능성 큼**
- Phase 1에서 템플릿 매칭 정확도를 실측한 뒤, **부족할 경우에만** YOLOv8 도입 (라벨링 비용 큼 — 섣불리 시작하지 않음)

**순목 / 테다시·쯔모기리 추적**
- 순목(turn): 자신의 버림패 매수 또는 전체 진행으로 카운트 — **이후 거의 모든 판단(리치·오시히키·위험패)의 전제**
- 테다시(手出し) / 쯔모기리(ツモ切り) 구분: 상대 텐파이 읽기의 핵심 신호. 작혼은 손에서 나오는 패와 쯔모한 패의 애니메이션이 다름 → 인식 난점이나 우선순위 높게 다룸

**출력**: `GameState` 객체 (아래 4.3)

### 4.3 Game State Model (`src/core/`)
```
GameState
├─ my_hand: list[Tile]            # 내 손패 (아카도라 플래그 포함)
├─ my_melds: list[Meld]           # 내 부로
├─ my_discards: list[Discard]     # 내 버림패 (후리텐 판정용)
├─ discards: dict[Seat, list[Discard]]   # 4인 버림패 (쯔모기리/테다시 구분)
├─ melds: dict[Seat, list[Meld]]  # 4인 부로 (울린 방향 포함)
├─ dora_indicators: list[Tile]    # 도라 표시패 (깡도라 포함)
├─ round: RoundInfo               # 장/국/본장, 내 자풍·장풍, 親/子
├─ scores: dict[Seat, int]
├─ riichi: dict[Seat, bool]
├─ turn: int                      # 순목
├─ tiles_left: int
└─ confidence: dict               # 요소별 인식 신뢰도
```
인식 신뢰도 필드를 함께 두어, 불확실한 인식은 UI에서 흐리게 표시.

### 4.4 Analysis Engine (`src/analysis/`)

**(a) 샹텐 / 우케이레 / 타패 추천** ★핵심
- `mahjong` 라이브러리로 일반형·치또이쯔·국사무쌍 샹텐 계산
- 우케이레: 보이지 않는 패 카운트(4 − 보이는 매수)로 각 후보패의 유효 매수 산출
- **타패 추천**: 각 후보 타패별로 → 우케이레 매수 + 대기 형태의 질(양면/간짱/변짱/단기) + 예상 타점 → 종합 점수로 "이 패를 버려라" 제시
- 텐파이 시 대기패·대기 형태 + **대기패 잔여 매수(벽·보이는 매수)** 표시

**(b) 후리텐 / 役なし 체크** ★핵심
- **후리텐**: 내 텐파이 대기패가 내 버림패에 하나라도 있으면 후리텐 → 빨강 경고 (론 불가)
- **役なし 텐파이**: 텐파이인데 역이 0개면 경고 ("형식 텐파이 — 리치/쯔모만 가능")

**(c) 역 추천 + 타점 추정**
- 현재 손패에서 도달 가능한 역 후보를 점수화: 핑후·탕야오·역패·이페코·산색·일기통관·찬타·혼일색·청일색·또이또이·치또이쯔 등
- 각 역의 "필요 패 / 우케이레 손실 / 기대 타점" 트레이드오프 제시
- 타점 추정: 도라 + **아카도라** 매수 반영, 예상 판/부

**(d) 위험패 분석**
- 트리거: 상대 리치 선언, 또는 부로·버림패 속도로 텐파이 추정
- 상대별로 내 손패 각 패의 안전도 등급:
  - **현물(genbutsu)**: 그 상대 버림패에 있음 → 100% 안전
  - **스지 세분화**: 중스지(中スジ) / 편스지(片スジ) / 우라스지(裏スジ) / 걸침스지(跨ぎスジ) 구분 — 뭉뚱그리지 않음
  - **노찬스 / 원찬스**: 보이는 패로 특정 양면 불가/거의 불가 판정 (벽 분석)
  - **字牌**: 보이는 매수로 안전도 (3장 보이면 단기만, 2장이면 비교적 안전)
  - **무스지 중장패**: 가장 위험. 도라·도라스지 추가 위험
  - **一発·리치 선언패 기준** 위험 패턴 가중
- 종합 위험도를 색상으로 오버레이 표시 (녹/황/적)

**(e) 오시히키(押し引き) 판단** ★핵심
- 내 패 가치(샹텐·타점·대기 질·점수상황) vs 방총 리스크(상대 추정 타점·위험패 보유) 비교
- 결론 제시: **밀어(押し)** / **접어(ベタオリ)** / **부분 후퇴**

**(f) 베타오리 시퀀싱**
- 접기로 결정 시, 매 순목 가장 안전한 패부터 순서대로 제시 (현물 → 스지 → 노찬스 → …)

**(g) 리치 판단**
- 리치 / 다마텐 / 안 침 — 대기 질(양면 여부), 타점, 순목, 점수상황 기준

**(h) 부로 판단**
- 폰·치 제안 시 → 쿠이탕·역패 확보·속도 이득 vs 멘젠 타점·방어력 손실 비교 → "울어/넘겨"

**(i) 상대 역 예측 (버림패 분석) — 보조지표**
- 버림패 시계열로 역 경향 추정. **확정 아닌 "경향"으로 UI 표기** (버림패만으론 본질적 신뢰도 한계)
  - **탕야오**: 초반 1·9·字牌 버림, 이후 중장패 유지
  - **혼일색 / 청일색**: 버림패가 1~2 수트에 편중
  - **역패**: 役牌 폰, 또는 초반 자풍·장풍패 비유지
  - **또이또이**: 폰 다수 + 슌쯔성 패 버림
  - **찬타·준찬타**: 중장패 위주 버림
  - **리치**: 리치봉/선언패로 확정
- 버림패 속도 + 테다시/쯔모기리 + 부로 수 + 순목으로 텐파이 시점 추정

**(j) 점수상황 / 착순 전략**
- 올라스 등 점수차 기반: 필요 타점 산출, 톱 목표 / 라스 회피 모드 전환 → 타패·리치·오시히키 판단에 가중

**(k) 리치 후 상태 모드**
- 내가 리치 선언 시 손패 고정 → 툴 역할 전환: 위험패 분석 + 쯔모 화료 판정만 표시 (타패 추천 비활성)

### 4.5 Overlay UI (`src/overlay/`)
- `PyQt6` 프레임리스 + `Qt.WA_TranslucentBackground` + `WindowStaysOnTopHint`
- `WS_EX_TRANSPARENT` 플래그로 클릭 통과(게임 조작 방해 안 함). 토글 키로 상호작용 모드 전환
- 작혼 창 위치를 추적하여 오버레이가 따라 이동
- 표시 패널:
  - **내 패 패널**: 샹텐, 타패 추천, 우케이레, 추천 역, 예상 타점, 후리텐/役なし 경고
  - **위험패 패널**: 손패 각 패의 안전도 색상 바, 오시히키 결론, 베타오리 추천 순서
  - **상대 패널 ×3**: 추정 역(경향), 텐파이 추정, 리치 여부
  - **상황 패널**: 도라·아카, 점수, 순목, 착순 전략 모드
- 패널은 드래그 이동·on/off·투명도 조절 가능. 설정은 YAML 저장

### 4.6 Main Loop (`src/core/`)
캡처 → 인식 → 상태 비교 → (변화 시) 분석 → UI 갱신. 캡처/인식/분석은 워커 스레드, UI는 메인 스레드(Qt 요구사항). 큐로 전달.

|------|------|--------|
| **Phase 0** | 환경 셋업, 캡처 PoC | 작혼 창 탐색 + 영역 캡처 동작 확인 |
| **Phase 1** | 내 패 인식 + 핵심 판단 | 손패 템플릿 매칭, 샹텐·우케이레, **타패 추천 / 후리텐 경고 / 役なし 경고**, 순목 추적, 아카도라. 수동 입력 fallback 포함. **여기서 템플릿 매칭 정확도 실측** |
| **Phase 2** | 버림패 인식 + 방어 판단 | 4인 버림패 인식, 위험패 분석(현물·스지 세분화·노찬스), **오시히키 판단 / 베타오리 시퀀싱**, 테다시/쯔모기리 구분 |
| **Phase 3** | 능동 판단 + 상대·상황 | **리치 판단 / 부로 판단**, 상대 역 예측(경향), 점수상황·착순 전략, 리치 후 모드, 도라·점수·멘젠 상태 |
| **Phase 4** | 정확도·완성도 | (필요시) YOLO 모델, 캘리브레이션 UI, PyInstaller 패키징 |

각 Phase는 독립적으로 사용 가능한 상태로 마무리(점진적 가치 제공).

## 7. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| **작혼 이용약관** | 화면 캡처 오버레이는 게임 메모리 조작이 아닌 회색지대이나, 자동화·외부도구 제한 조항 가능성 있음 | **개인 학습용으로 한정**, 공개 배포 전 약관 재검토. 입력 자동화(자동 클릭)는 포함하지 않음 |
| 패 인식 정확도 | 애니메이션·3D 원근·패 겹침으로 오인식 | Phase 1은 평면인 내 손패만. 인식 신뢰도 표시 + 수동 보정 입력 fallback. Phase 1서 정확도 실측 후 필요시 ML |
| 클라이언트 업데이트 | UI 좌표·패 그래픽 변경 시 인식 깨짐 | 좌표를 상대좌표 프로파일로 외부화, 캘리브레이션 UI로 사용자 재보정 가능 |
| 전체화면 독점 모드 | 오버레이가 안 보임 | 작혼을 창 모드로 실행하도록 안내. 창 모드 자동 감지·경고 |
| 테다시/쯔모기리 인식 | 상대 텐파이 읽기 정확도 직결, 애니메이션 차이 미세 | Phase 2 별도 과제로 다룸. 실패 시 "쯔모기리 추정" 신뢰도 낮춰 표기 |
| 상대 역 예측 신뢰도 | 버림패만으론 본질적 한계 | "확정" 아닌 "경향" 보조지표로 명시. 과신 유도 UI 금지 |
| 성능 | 캡처·인식 부하 | dxcam GPU 캡처, 프레임 변화 감지 시에만 분석, fps 제한 |
| 개발 환경 | 개발은 Linux, 대상은 Windows | 캡처·오버레이 코드는 Windows에서만 실행·검증 가능. 로직(분석 엔진)은 OS 무관 — 단위 테스트로 커버 |
